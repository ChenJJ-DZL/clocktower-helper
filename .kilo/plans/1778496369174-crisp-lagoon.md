# 洗衣妇（Washerwoman）能力实现计划

## 现状分析

当前 `washerwoman.ability.ts` 存在以下问题：

1. **preCheckAliveAndStatus 内联定义**：与其他角色文件重复，未使用 `roleAbility.types.ts` 中的 `commonPreCheckAlive`
2. **calculate 逻辑不完整**：
   - 仅依赖 `meta.initialNightInfo?.washerwomanInfo`，无自动回退（auto-generation）
   - 无 Spy 互动处理（间谍可注册为镇民）
   - 无 Recluse 互动处理（陌客可不注册为镇民）
   - Drunk/Poisoned 的假信息生成随机性过大，可能暴露醉酒/中毒状态（应仍返回镇民角色）
3. **stateUpdate 为空**：未记录信息供后续游戏使用
4. **postProcess 仅 console.log**：未生成说书人 UI 文案
5. **字段名不匹配**：模板中的 `roleName`/`firstNight`/`otherNight` 与 `IRoleAbility` 接口不一致

## 核心架构约束

- 使用 `createRoleAbility` 工厂函数（来自 `../core/roleAbility.types`）
- Pipeline 顺序：preCheck → calculate → stateUpdate → postProcess
- `IRoleAbility` 接口必须满足：`roleId`, `abilityId`, `abilityName`, `triggerTiming`, `wakePriority`, `firstNightOnly`, `wakePromptId`, `targetConfig`, `preCheck[]`, `calculate[]`, `stateUpdate[]`, `postProcess[]`

## 实现方案

### 1. Import 结构调整

```
保留: createRoleAbility, AbilityTriggerTiming 来自 '../core/roleAbility.types'
保留: MiddlewareContext 来自 '../../utils/middlewarePipeline'

不再内联 preCheckAliveAndStatus → 改用 commonPreCheckAlive 来自 '../core/roleAbility.types'
```

### 2. preCheck 阶段

使用 `commonPreCheckAlive` 中间件（已在 `roleAbility.types.ts` 第 18-41 行定义）：
- 检查玩家是否存活
- 标记 `isDrunk`/`isPoisoned`/`abilityEffective`
- 不再重复定义内联版本

### 3. calculate 阶段

核心逻辑（单个 middleware 函数 `washerwomanCalculate`）：

```
1. 获取 abilityEffective 状态
2. 若已由更高级 middleware（abilityPriorityCalculation）判定为 false → 生成假信息
3. 若正常生效：
   a. 优先检查 context.storytellerInput（说书人手动覆盖）
   b. 其次检查 meta.initialNightInfo?.washerwomanInfo（系统预设信息）
   c. 若以上均无，自动生成：随机选一位存活镇民 + 另一位存活非自身的玩家
4. 假信息生成规则（能力无效时）：
   a. 仍返回镇民角色（避免暴露醉酒/中毒）
   b. 从剧本镇民中随机选
   c. 从存活玩家中随机选两位
5. 角色互动处理：
   a. Spy：注册为镇民角色时，可作为 "镇民" token 目标（洗衣妇会得知 Spy 扮演的角色）
   b. Recluse：可能不注册为镇民，从而影响目标选择
   c. Drunk 玩家：若 Drunk 是洗衣妇本人由 preCheck 处理；若 Drunk 是目标玩家，按其自以为的角色注册
6. 输出: { seat1: string; seat2: string; roleName: string }
```

### 4. stateUpdate 阶段

记录信息到 `context.snapshot` 中，使用游戏通用机制：
```
- 记录 washerwomanInfo 到 seat 的 meta 中
- 标记当前座位 hasReceivedInfo = true（防重复使用）
- 若有能力相关的全局状态变更（如启用旗帜标记等），在此处理
```

### 5. postProcess 阶段

生成说书人友好输出：
```
- 填充 wakePromptId 所需的模板变量（seatNo, seat1, seat2, roleName）
- 输出 detailedSimulationLog 包含：
  * 当前玩家角色
  * 两名目标玩家 seatNo + 角色名
  * 展示的角色名称
  * 信息是否受醉酒/中毒/涡流影响
  * 是否有 Spy/Recluse 等互动干扰
- 生成玩家看到的文案（what player sees）
```

### 6. 配置字段确认

### 7. wakePromptId 使用

promptDictionary.ts 中已有的模板：
- `role.washerwoman.wake`: `请唤醒洗衣妇（{{seatNo}}号玩家），告知其{{seat1}}号和{{seat2}}号玩家中有一名是{{roleName}}`
- `ability.washerwoman.info`: `洗衣妇获得信息：{{seat1}}号和{{seat2}}号玩家中有一名是{{roleName}}`

postProcess 中使用这些模板生成界面文案。

## 验证方式

1. TypeScript 编译检查：`npx tsc --noEmit` 或 `npm run typecheck`
2. 确认与现有角色（chef, librarian, investigator）的实现风格一致
3. 确认 calculate 逻辑覆盖以下场景：
   - 正常首夜：返回两名玩家 + 镇民角色
   - 醉酒/中毒：返回合理的假信息（镇民角色，但目标/角色可能错误）
   - Spy 在目标中：Spy 可被当作镇民展示
   - Recluse 可被当作非镇民从而不进入候选
   - 说书人手动覆盖：优先使用 storytellerInput
   - 极端情况（5人局+男爵）：无镇民时返回洗衣妇自己
