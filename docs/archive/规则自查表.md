# 血染钟楼说书人助手 - 规则自查表

## 概述
本报告基于对 `json/rule/` 文件夹中所有规则文件的详细分析，以及与当前项目代码实现的对比，识别出规则与实现之间的差异。

**分析时间**: 2026年4月1日  
**规则文件版本**: 官方血染钟楼规则  
**项目版本**: 当前项目实现

## 一、主要规则文件分析总结

### 1.1 规则概要 (规则概要.json)
- 游戏规模：5-20名玩家
- 阵营：善良 vs 邪恶
- 游戏环节：白天（提名、投票、处决）、夜晚（能力行动）
- 获胜条件：
  - 善良阵营：处决恶魔
  - 邪恶阵营：恶魔存活到只剩下3名玩家，或特定条件达成

### 1.2 重要细节 (重要细节.json)
- **处决规则**：每天只能处决一名玩家，需要获得超过半数存活玩家的投票
- **死亡玩家**：可以继续发言，但不能投票（除非有特殊能力如"幽灵票"）
- **醉酒/中毒**：玩家会获得错误信息，但不知道自己是醉酒/中毒状态
- **说书人裁量权**：说书人可以调整规则以保持游戏平衡和趣味性
- **私聊与笔记**：允许玩家私聊，说书人和玩家都可以做笔记

### 1.3 夜晚行动顺序
- **首夜**：87个行动步骤
- **其他夜晚**：118个行动步骤

### 2.2 处决规则实现分析

#### 2.2.1 符合规则的部分
1. **每天只能处决一名玩家**：在 `src/hooks/useExecutionHandlers.ts` 的 `executeJudgment` 函数中，通过 `todayExecutedId !== null` 检查实现
2. **需要多数投票**：通过 `threshold = Math.ceil(aliveCount / 2)` 计算半数
3. **平票处理**：当最高票数并列时，处理为平安日无人被处决

#### 2.2.2 不符合规则的问题
1. **阈值计算错误**：
   - 规则要求：需要"超过半数"存活玩家的投票
   - 当前实现：`Math.ceil(aliveCount / 2)` 对于偶数玩家数计算错误
   - 示例：6名存活玩家需要4票（>3），但代码要求3票（ceil(6/2)=3）
   - 影响：可能导致少数派通过投票执行处决

2. **死亡玩家投票处理**：代码中允许死亡玩家投票（通过 `hasGhostVote`），但规则中死亡玩家通常不能投票，除非有特殊能力

### 2.3 投票规则实现分析

#### 2.3.1 符合规则的部分
1. **投票阈值计算**：基于存活核心玩家数量计算半数阈值
2. **恶魔投票标记**：支持恶魔投票的特殊标记
3. **投票合法性验证**：检查票数是否为自然数，不超过玩家总数

#### 2.3.2 特殊投票能力未完全实现
1. **管家 (Butler)**：规则中管家必须跟随主人投票，项目中有 `butler.ability.ts` 但需要验证投票逻辑完整性
2. **狂热者 (Zealot)**：规则中当存活玩家≥5时，狂热者必须在所有提名中投票，项目中未找到强制投票实现
3. **报丧女妖 (Banshee)**：规则中有双票能力，项目中描述存在但未找到投票逻辑实现
4. **维齐尔 (Vizier)**：规则中有强制处决能力，项目中为 `hidden` 占位，未实现投票强制通过逻辑

### 2.4 角色能力实现差异

#### 2.4.1 缺失或部分实现的角色
1. **失忆者 (Amnesiac)**：存在角色文件但缺失于夜晚行动顺序
2. **魔术师 (Magician)**：仅在交叉规则引用中出现，可能无独立文件
3. **军团 (Legion)**：投票规则特殊（只有邪恶投票则记0票），项目中为占位实现
4. **维齐尔 (Vizier)**：公开存在，可能影响投票/处决，项目中为 `hidden` 占位
5. **街头风琴手 (Organ Grinder)**：影响投票UI/计票流程，项目中为占位不改变现有投票逻辑

#### 2.4.2 能力交互规则
规则文件中的相克规则、隐性规则等可能未在代码中完全实现，需要进一步检查角色间的特殊交互。

### 2.5 UI映射与游戏流程

#### 2.5.1 游戏阶段转换
**规则要求**：白天→黄昏→夜晚→黎明的完整循环  
**实现检查**：已正确实现，`useGameFlow.ts` 中的 `handleDayEndTransition` 正确处理了白天结束后的转换逻辑，符合"处决后直接入夜"规则

#### 2.5.2 玩家状态显示
**规则要求**：死亡玩家应有明确标识，但可继续参与讨论  
**实现检查**：UI中正确显示玩家状态（存活/死亡/醉酒/中毒等）

#### 2.5.3 私聊与笔记功能
**规则允许**：玩家间私聊，说书人和玩家做笔记  
**实现检查**：项目中有笔记功能（`NoteEditModal`），私聊功能可能需要进一步实现

### 2.6 邪恶阵营胜利条件实现

## 五、结论

当前项目在核心游戏机制方面有较好的实现，但与官方规则文件存在以下主要差异：

1. **夜晚行动顺序不完整**：缺少多个角色，顺序不一致
2. **处决阈值计算错误**：对于偶数玩家数，需要超过半数但当前计算为半数
3. **特殊投票能力未完全实现**：管家、狂热者、报丧女妖、维齐尔等
4. **部分角色能力缺失或部分实现**

建议按照优先级顺序进行修改，首先修复夜晚行动顺序和处决阈值计算，然后逐步完善特殊投票能力和缺失角色。

通过系统性的修改和测试，可以使项目更贴近官方规则，提供更准确和完整的血染钟楼游戏体验。
