# 血染钟楼夜晚逻辑重构架构书

## 一、现有代码设计缺陷分析

### 1. 职责耦合严重
`useNightLogic.ts` 作为核心Hook，同时承担了 **队列生成、状态管理、技能结算、UI交互、日志记录** 等多达12种不同职责，违反单一职责原则，代码量超过1000行，维护成本极高。

### 2. 夜晚顺序管理混乱
- 官方规则（`夜晚行动顺序.json`）与代码实现完全脱节，顺序硬编码在各个角色定义文件中
- 存在两套独立的顺序系统：角色定义内的`firstNightOrder/nightOrder`和全局覆盖表`nightOrderOverrides.ts`，容易冲突
- 动态过滤逻辑（如送葬者无人处决时不唤醒）分散在多个地方，没有统一的规则引擎

### 3. 提示词体系缺失
- 说书人指引文本分散在各个角色实现、模态框组件、日志函数中，没有统一管理
- 相同逻辑的提示词存在多处重复，不一致性高，无法支持国际化和动态调整
- 缺乏模板能力，无法根据游戏状态动态生成个性化指引

### 4. 技能结算时序脆弱
- 当前实现依赖函数调用顺序，没有统一的状态机管理，"夜晚行动过时不候"等隐性规则的处理存在漏洞
- 事件驱动缺失：技能结算、状态变更、UI更新之间没有解耦，一处修改可能影响多个模块
- 不可变状态保证不足：存在多处直接修改seat属性的行为，容易产生难以追踪的副作用

### 5. 可扩展性差
- 新增角色需要修改至少3个文件（角色定义、夜晚逻辑、提示词），不符合开放封闭原则
- 没有统一的扩展点，特殊角色（如亡骨魔、普卡）的逻辑硬编码在主流程中

## 二、JSON规则到控制台流程的映射方案

### 1. 映射架构
```
官方JSON规则 → 解析层 → 规则引擎 → 动态队列生成 → 控制台流程执行
```

### 2. 实现步骤
1. **静态解析层**：
   - 新增 `nightOrderParser.ts` 统一解析 `josn/夜晚行动顺序.json`，生成标准化的角色顺序映射表
   - 自动同步官方顺序到角色定义，消除硬编码和覆盖表的冲突
   - 支持多剧本自动切换：根据当前选择的剧本自动加载对应夜晚顺序

2. **动态规则引擎**：
   - 统一处理所有动态过滤规则（如送葬者唤醒条件、死亡角色保留能力条件等）
   - 内置官方隐性规则："过时不候"、"首夜特殊顺序"、"死亡角色唤醒例外"等
   - 支持Jinx规则动态调整顺序：当存在相克角色时，自动调整唤醒顺序

3. **控制台流程映射**：
   - 每个队列项自动对应3个控制台步骤：`唤醒→操作指引→结算确认`
   - 自动生成流程进度条，显示当前进度和后续步骤
   - 支持暂停、回滚、跳过等操作，满足说书人灵活调整需求

## 三、说书人指引（Prompt）数据流重构

### 1. 三层架构
```
提示词字典层 → 模板引擎层 → 渲染层
```

### 2. 实现方案
1. **集中化字典层**：
   - 新增 `promptDictionary.ts` 统一管理所有提示词，分类存储：
     - 通用系统提示（入夜、天亮、结算等）
     - 角色操作提示（每个角色的唤醒语、操作指引）
     - 结果反馈提示（成功、失败、特殊效果等）
   - 支持多语言扩展，预留国际化接口
   - 与角色定义绑定，新增角色时自动关联对应提示词

2. **智能模板引擎**：
   - 新增 `promptEngine.ts` 实现变量替换、条件判断、循环渲染能力
   - 支持动态参数注入：`{{seatNo}}`、`{{roleName}}`、`{{targetSeatNo}}`等
   - 内置规则判断：根据醉酒、中毒、死亡等状态自动调整提示内容
   - 支持说书人自定义提示模板覆盖

3. **渲染层**：
   - 统一输出到游戏控制台、模态框、日志系统三个终端
   - 支持富文本格式：高亮、加粗、颜色标记关键信息
   - 自动记录所有提示历史，方便回溯

## 四、技能结算架构设计

### 1. 设计模式选择：**状态机+事件总线+中间件管道**
三层架构实现完全解耦：
```
事件总线 → 状态机 → 中间件管道 → 技能结算
```

### 2. 各层职责
1. **事件总线（GameEventBus）**：
   - 强类型事件系统，所有状态变更、技能触发、UI操作都通过事件传递
   - 支持事件溯源，所有操作可回溯、可重放
   - 实现发布-订阅模式，完全解耦各模块通信

2. **有限状态机（NightStateMachine）**：
   - 定义夜晚所有合法状态：`idle→queuing→waking→processing→confirming→dawn`
   - 内置状态转移规则，禁止非法状态跳转
   - 统一管理夜晚全局状态：当前步骤、队列、已结算技能等

3. **中间件管道**：
   - 技能结算前依次经过多个中间件处理：
     ```
     技能触发 → 醉酒/中毒检测 → Jinx规则检测 → 能力有效性校验 → 效果结算 → 状态更新 → 日志记录
     ```
   - 每个中间件独立实现，可单独测试、扩展
   - 支持动态插入自定义中间件，满足特殊剧本需求

4. **技能结算层**：
   - 每个角色技能独立实现为纯函数，输入为当前状态快照，输出为新的状态快照
   - 严格遵循不可变性原则，所有状态变更返回新对象
   - 内置原子性保证：技能结算要么完全成功，要么完全回滚

### 3. 实时结算保证
- 采用不可变快照模式：每一步结算都生成完整的状态快照
- 异步技能自动排队：支持延迟结算效果（如普卡中毒、女巫诅咒等）
- 冲突解决机制：多个效果同时触发时，按照官方规则优先级自动排序
- 一致性校验：结算后自动校验状态合法性，发现异常自动回滚并告警

## 五、重构阶段划分
### 阶段一（当前）：基础架构搭建
- 完成核心模块开发：nightOrderParser、promptDictionary、promptEngine、gameEventBus
- 零TS错误、零循环依赖

### 阶段二：旧逻辑迁移
- 逐步迁移现有夜晚逻辑到新架构
- 保持兼容，不影响现有功能
- 配套完整测试用例

### 阶段三：功能增强
- 实现官方规则全量覆盖
- 支持动态剧本加载
- 完善说书人控制台功能

### 阶段四：性能优化
- 实现结算缓存
- 支持大规模游戏（>30人）流畅运行
- 增加结算校验和容错机制
