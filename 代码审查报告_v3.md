# 血染钟楼助手 (Clocktower Helper) 全面代码审查报告 v3

## 1. 架构审查 (Architecture Review)

### 1.1 核心状态管理 (State Management)
- **现状**: 采用了 `GameContext` + `useGameActions` 的集中式管理。
- **优点**: 保证了单一数据源，通过 `useRef` (如 `currentWakeIndexRef`) 巧妙解决了 React 闭包导致的异步状态一致性问题。
- **改进点**: `useGameController.ts` 文件已超过 2800 行，业务逻辑与 UI 控制耦合过紧。建议按阶段（设置阶段、夜晚阶段、白天阶段）拆分为专用的 Controller Hook。

### 1.2 角色系统演进 (Role System)
- **现状**: 正在从基于属性的旧系统 (`app/data.ts`) 迁移到 OOP 角色定义系统 (`src/roles`)。
- **架构评估**: 现在的 `RoleDefinition` 接口设计非常完善，涵盖了从首夜顺序、夜晚行动到处决回调 (`onExecution`) 的全生命周期。
- **风险**: 系统处于“混血”状态。很多逻辑在 `nightLogic.ts` 中针对旧 Role 对象编写，而新角色通过 `getRoleDefinition` 获取逻辑。

## 2. 规则与逻辑审查 (Rules & Logic Review)

### 2.1 游戏规则覆盖率
- **核心角色**: 《麻烦滋事》的角色逻辑基本成熟。
- **复杂角色**: 《紫罗兰》和《黯月初升》的部分角色（如物理学家、僵怖）虽然已有 Class 定义，但 `handler` 仍标为 `undefined` (TODO)。
- **规则准确性**: 
    - 死亡判定 (`killPlayer`) 已经过重写，支持了免死、保护、假死等复杂链式反应。
    - 胜负判定 (`checkWinCondition`) 优先判定金字塔结构合理。

### 2.2 逻辑漏洞 (Logic Leaks)
- **发现**: 部分 UI 组件仍存在直接操作 `Seat` 属性的情况，而非通过 `dispatch` 动作。
- **建议**: 严禁在组件层直接修改 `seats[i].isDead = true`，必须统一调用 `killPlayer` 动作以触发连带反应（如红唇、恶魔传位等）。

## 3. 废弃代码与技术债 (Deprecated Code & Technical Debt)

### 3.1 废弃字段清单
以下字段在 `app/data.ts` 中已被标记为 `@deprecated`，但仍有大量存量代码在使用：
- `firstNightOrder`, `otherNightOrder`: 建议逐步迁移至 `RoleDefinition.night.order`。
- `nightActionType`, `nightActionAmount`: 建议迁移至 `NightTargetConfig`。

### 3.2 待办事项 (TODOs)
- **角色迁移**: 超过 20+ 角色的行动逻辑尚未从旧的 `nightLogic.ts` 迁移到 OOP Handler 中。
- **描述补全**: 许多新角色文件的 `detailedDescription` 和 `clarifications` 为空。

## 4. 部署与环境 (Deployment & Environment)

### 4.1 技术栈更新
- **前端**: 已升级至 **Next.js 15.1.6** 和 **React 19**，并使用了 **TailwindCSS 4**。性能表现优异。
- **部署**: 完美适配 Vercel，支持 Edge Runtime 优化。

### 4.2 测试状态
- **测试体系**: 提供了纯逻辑层的强力支撑（Jest）以及跨角色的 E2E 模拟（Playwright）。
- **优化空间**: E2E 测试仍存在少量因 `sleep` 引起的非确定性失败，建议优化为 `waitForSelector` 等被动触发。

## 5. 综合评价与建议

**评价**: 项目正处于向高模块化架构转型的关键期。核心逻辑鲁棒性很高，能够处理《血染钟楼》中极其复杂的技能连锁。

**接下来的行动路径**:
1. **清理废弃字段**: 在确保 OOP 迁移完成的角色中，强制移除对 Role 对象旧 order 字段的依赖。
2. **拆分 Controller**: 将 `useGameController` 按功能拆分，减少单个文件的维护成本。
3. **补全角色 Handler**: 优先完成当前剧本中角色的 `handler` 迁移。
